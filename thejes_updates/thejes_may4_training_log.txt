TRAINING DETAILS
----------------
Method          : QLoRA — 4-bit NF4 quantization + LoRA adapters merged into full model
GPU             : NVIDIA H100 80GB HBM3 (RunPod On-Demand, $2.99/hr)
Batch size      : 4 (auto-detected for 80GB GPU), grad accumulation 4
LoRA rank       : r=16, alpha=16
Optimizer       : paged_adamw_8bit
Scheduler       : cosine
Max seq length  : 1024 (packing enabled)

Coding dataset  : 60,000 samples
  - CodeFeedback-Filtered-Instruction : 40,000 samples
  - CodeAlpaca-20k                    : 12,000 samples
  - Python Instructions 18k           : 8,000 samples

Meeting dataset : ~22,000 samples
  - DialogSum (knkarthick)
  - MeetingBank (huuuyeah)
  - QMSum (yale-nlp)

Mixtral coding  : 177 steps | ~24 min | Loss: 0.5019
Mixtral meeting : ~31 min   | Loss: 1.7773
Gemma models    : ~1.5 hrs each (running overnight)

ISSUES SOLVED
-------------
1. HF model cache filling root disk (20GB container)
   Fix: export HF_HOME=/workspace/hf_cache

2. Root disk 100% full during push_to_hub
   Fix: rm -rf /root/.cache/huggingface (freed 14GB)
         export TMPDIR=/workspace/tmp
         Used huggingface-cli upload directly from saved merged folder

3. tmux/screen not available on RunPod
   Fix: nohup + tail -f for live monitoring with background persistence

4. Gemma 2 "System role not supported" error
   Fix: Modified format_sample() in train.py to detect gemma_mode=True
        and prepend system prompt to the first user message instead of
        using a separate system role

5. train.py only supported coding/meeting (not main or --base flag)
   Fix: Added --base (qwen/mixtral/gemma) and --model main arguments

6. Disk space running out mid-training (Mixtral 93GB + merged 25GB)
   Fix: gemma_auto.sh deletes Mixtral cache and old merged models
        before starting Gemma training
